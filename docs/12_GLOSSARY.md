# 12 — Glossary

## 1. Terms & Concepts

### Access Token Cookie
A short-lived (15-minute) httpOnly cookie named `access_token` containing the signed JWT. Set by `POST /auth/login` and `POST /auth/register`. Read automatically by the browser on every request — never manually attached in frontend code.

### Active Session
A record in the `active_sessions` database table representing one authenticated login instance on one device. Created/updated by `fn_refresh_session()`. Revoked via `fn_revoke_session()` or `DELETE /auth/sessions/other`.

### Board Column
A `board_columns` table record defining a stage in a Kanban workflow (e.g., "To Do", "In Progress", "Review", "Done"). Each column has a `column_type` enum (`TODO`, `IN_PROGRESS`, `REVIEW`, `DONE`) used for KPI aggregation.

### Canonical View Layer
PostgreSQL views named `v_*_canonical` (e.g., `v_tasks_canonical`, `v_dashboard_kpis_canonical`) that serve read requests from the backend without exposing raw table implementations. Application code **must only** read from these views — never from base tables directly.

### Dashboard KPIs
Organization-wide Key Performance Indicators exposed by `v_dashboard_kpis_canonical`: total tasks, tasks by status (todo/in-progress/review/done), overdue tasks, total boards, team size, pending proposals count, active meetings count.

### Extracted Task Proposal
A draft task item generated by LLM analysis of a completed meeting transcript. Stored in the `task_proposals` table with status `pending`. Awaits Manager/Superadmin review in the proposal queue before being converted into a Kanban task card via `fn_approve_task_proposal`.

### httpOnly Cookie Auth
KAIO's authentication mechanism. The backend sets `access_token` (15 min) and `refresh_token` (7 days) as httpOnly cookies on login/register. The frontend Axios client sends them automatically via `withCredentials: true`. This prevents JavaScript XSS token theft vs. localStorage-based auth.

### Invitation
A record in `organization_invitations` created when a Manager/Superadmin invites a new user by email. Contains a secure token, target role, expiry time, and `accepted_at` timestamp. Can be **revoked** (deleted) while still pending via `DELETE /invitations/{id}`.

### Meeting Session
An active or historical video meeting instance managed by KAIO. Identified by a unique `session_id` string. Stored in the `meeting_sessions` table scoped to an `org_id`.

### Participant Presence Timeline
A chronological log (`ParticipantPresenceTimeline`) recording participant join events, leave events, display name edits, and host transfers captured by the Chrome Extension's DOM observer during a meeting session.

### Playwright Bot Controller
The browser automation layer (`app/meeting/bot/controller/`) that launches Chromium, logs into Google services, navigates to meeting URLs, and accepts meeting permissions.

### MediaRecorder Script
A custom JavaScript module (`capture_script.js`) injected into the Playwright page context that captures tab audio via Web Audio API and streams WebM chunks back to Python.

### RBAC (Role-Based Access Control)
KAIO implements three user roles:
- `MEMBER` — can create, edit, move assigned tasks, view boards, comment.
- `MANAGER` — all Member permissions + invite users, review/approve task proposals, access Manager Dashboard, start meetings, manage project settings, revoke invitations.
- `SUPER_ADMIN` — all Manager permissions + admin panel (user/board CRUD), delete users, change any role.

### Refresh Token Cookie
A long-lived (7-day) httpOnly cookie named `refresh_token` scoped to path `/api/v1/auth`. Used by `POST /auth/refresh` to issue new access/refresh cookie pairs without re-entering credentials.

### RequireRole Guard
A React Router route guard (`src/routes/RequireRole.tsx`) that wraps routes requiring specific roles. Reads `user.role` from `authStore` and redirects unauthorized users to `/dashboard`.

### Security Event
A record in `security_events` created by `fn_log_security_event()` for authentication-relevant actions: `user_login`, `user_logout`, `session_revoked`, `password_changed`, `role_updated`. Includes IP address and User-Agent metadata.

### Speaker Attribution
The process executed by `DynamicAttributionEngine` that correlates synthetic speaker labels from Deepgram diarization (e.g., `speaker_0`) with real participant identities from the presence timeline, producing a confidence-scored assignment.

### Speaker Turn
A discrete audio segment identified during Deepgram diarization, containing a start timestamp, end timestamp, and a synthetic integer speaker label (e.g., `speaker_0`).

### Stored Function (fn_*)
A PostgreSQL PL/pgSQL function named with the `fn_` prefix that encapsulates all data mutation logic. Backend Python code calls these exclusively for writes — never inline SQL.

### Task Proposal Approval Queue
The interface (backend: `task_proposals.py` router + DB `task_proposals` table; frontend: proposal drawer components) through which Managers and Superadmins review, edit, approve, or reject AI-extracted task proposals from meeting transcripts.

### Timesheet
A weekly effort log record (`timesheets` table, `v_timesheets_canonical` view) representing 7 days of time entries for a user starting on an organization-configured `week_start_day`. Has lifecycle statuses: `draft`, `submitted`, `approved`, `rejected`, `recalled`.

### Timesheet Entry
A daily time log row (`timesheet_entries` table, `v_timesheet_entries_canonical` view) recording hours logged against a specific project board, task card, or general work type (`task`, `meeting`, `general`, `leave`, `holiday`).

### Timesheet Policy
Organization-wide configuration settings (`timesheet_policies` table, `v_timesheet_policy_canonical` view) governing time logging rules: `week_start_day`, `standard_hours_per_day`, `standard_hours_per_week`, `max_hours_per_day`, `overtime_policy`, `submission_deadline_days`, `allow_future_entry`, `allow_past_entry_days`, `require_task_link`, and `allow_member_recall`. Managed by Superadmins via `TimesheetPolicyForm`.

### Timesheet Approver Assignment
A record (`timesheet_approver_assignments` table, `v_timesheet_approver_assignments_canonical` view) designating a Manager or Superadmin as an active timesheet reviewer for submitters within an organization.

### Zustand Store
A lightweight global state container (`zustand` v5) used throughout the KAIO frontend. Replaces React Context for all global state. There are 10 stores: `authStore`, `boardStore`, `taskStore`, `adminStore`, `notificationStore`, `organizationStore`, `preferencesStore`, `projectSettingsStore`, `activityStore`, `uiStore`.
